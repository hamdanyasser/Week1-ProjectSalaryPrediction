# 1. Project identity

- Project name: `PayScope`
- One-sentence description: A local-first salary prediction app that trains a Decision Tree on the DS Salaries dataset, serves predictions through FastAPI, and explains them through a polished Streamlit dashboard.
- Main purpose of the app: Help a user understand salary patterns in the dataset and get a live salary estimate with plain-English context for why the estimate makes sense.

# 2. Assignment requirements we are trying to satisfy

- Salary Prediction Application
- Decision Tree model
- GET API for predictions
- separate Python script that calls the API with proper error handling
- local LLM via Ollama
- Supabase persistence
- Streamlit dashboard
- storytelling is a major grading factor
- no vibe coding
- code must be explainable in review

# 3. Current project strategy

- The chosen strategy is local-first: get training, API, client script, and Streamlit working locally before relying on cloud services.
- The dashboard is story-first: the visual flow matters more than adding extra backend complexity.
- The backend is intentionally small and code-review-safe. Most business logic lives in a few plain Python files instead of a large package structure.
- Friday’s audience is mostly non-technical, so the dashboard avoids technical jargon and focuses on quick visual understanding.
- Saturday’s code review requires understanding every important line, so the project avoids clever abstractions and favors explicit logic.
- The dashboard is intended to feel polished and product-like, with a hero section, KPI cards, chart cards, captions, a standout prediction section, and clear takeaways.
- The final architecture direction still supports Supabase-backed dashboard reads later, but the local FastAPI -> Streamlit flow is the first-class path right now.

## Core files

- `config.py`  
  Central settings loader. Reads `.env`, resolves paths, exposes `Settings`, and defines whether Ollama or Supabase are enabled.

- `ml.py`  
  Core ML module. Loads raw data, cleans it, builds the preprocessing + `DecisionTreeRegressor` pipeline, trains it, saves artifacts, and runs predictions.

  Prepares dashboard summaries, KPI values, chart-ready grouped data, takeaways, and peer-group explanation data used after prediction.

- `train_model.py`

- `predict_client.py`

# 5. Implemented functionality

- Model training is implemented in `ml.py` and triggered by `train_model.py`.
- The model is a scikit-learn pipeline with:
  - `OneHotEncoder(handle_unknown="ignore")` for categorical features
  - passthrough for `remote_ratio`
  - `DecisionTreeRegressor(max_depth=12, min_samples_leaf=4, random_state=42)`
- Data cleaning is implemented and currently produces `data/processed/cleaned_salaries.csv`.
- Saved artifacts already exist:
  - `artifacts/decision_tree_pipeline.joblib`
  - `artifacts/model_metrics.json`
- FastAPI is implemented in `api.py` with:
  - `GET /health`
  - `GET /predict`
- `GET /predict` validates:
  - `experience_level` as `EN | MI | SE | EX`
  - `employment_type` as `FT | PT | CT | FL`
  - `company_size` as `S | M | L`
  - `employee_residence` and `company_location` as 2-letter codes
  - `remote_ratio` as `0`, `50`, or `100`
- Prediction flow is implemented as:
  - request comes into FastAPI
  - inputs are normalized
  - model artifact is loaded
  - prediction is generated
  - peer-group explanation is built from the cleaned dataset
  - optional Supabase save runs if env vars are configured
- The API response includes:
  - `predicted_salary_usd`
  - `normalized_inputs`
  - `model_name`
  - `peer_context`
- Dashboard sections currently implemented in `dashboard.py`:
  - hero / framing
  - KPI snapshot
  - salary distribution
  - experience vs salary
  - employment type comparison
  - top roles by median salary
  - remote/work style comparison
  - salary spread by role
  - prediction studio
  - why this prediction makes sense
  - final takeaways
  - optional saved history from Supabase
- The separate Python client script is implemented and handles:
  - timeout
  - connection failure
  - HTTP errors
  - invalid JSON
- Tests exist and currently pass:
  - `4 passed` during inspection
- README exists and reflects the local run flow plus optional Ollama and Supabase notes.
- Ollama support is implemented inside `dashboard.py`.
- Supabase-related code exists in two places:
  - write to Supabase from `api.py`
  - read prediction history from Supabase in `dashboard.py`
- Supabase behavior is gated by env vars and is **not verified** in this repo because the current `.env` leaves Supabase fields blank.

# 6. Run flow

## From scratch

1. Install dependencies:

```bash
python -m pip install -r requirements.txt
```

2. Train the model and generate local artifacts:

```bash
python train_model.py
```

3. Start FastAPI:

```bash
uvicorn api:app --reload --host 127.0.0.1 --port 8000
```

4. Start Streamlit in a second terminal:

```bash
streamlit run dashboard.py --server.address 127.0.0.1 --server.port 8501
```

5. Optional: run the separate client script:

```bash
python predict_client.py --experience-level SE --employment-type FT --job-title "Data Scientist" --employee-residence US --company-location US --company-size M --remote-ratio 100
```
